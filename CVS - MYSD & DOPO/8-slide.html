S2: BUENAS PRÁCTICAS

Buenas Prácticas para Proyectos por Pares

Reglas de Juego
ESTRUCTURA

Rama por Integrante

Crea una rama con tu Apellido para aislar tus cambios. Nunca trabajes directo en main.

REGLA DE ORO

Main es Sagrada

La rama principal debe tener siempre código estable. Solo se actualiza vía Pull Request.

SINCRONIZACIÓN

Pull Frecuente

Antes de empezar, haz git pull origin main para evitar conflictos grandes después.

CALIDAD

Revisión por Pares

Todo cambio requiere la aprobación de tu compañero (Reviewer) antes de fusionarse.

HISTORIAL

Commits Descriptivos

Usa verbos imperativos. Ej: "feat: crear login" en vez de "cambios".

Ejemplo: ❌ "arreglando cosas" ✅ "fix: corregir error en conexión DB"
En la Práctica
bash — flujo ideal
# 1. Crear rama personal git checkout -b sanchez-feature # 2. Guardar cambios (semántico) git add . git commit -m "feat: implementar login" # 3. Subir a la nube git push origin sanchez-feature
LO QUE NO DEBES SUBIR

Archivos compilados (.class, .exe)

Carpetas IDE (.vscode, .idea)

Contraseñas o credenciales

Usa .gitignore para excluir esto automáticamente.
