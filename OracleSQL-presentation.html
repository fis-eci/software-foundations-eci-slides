Tipos de Pruebas en DOPO

Caso Nominal

El "Camino Feliz" (Happy Path). Entradas válidas donde todo funciona según lo esperado y devuelve un resultado correcto.

Ejemplo

// Factorial de 5 es 120
factorial(5) → 120

Caso de Borde

Valores límite o extremos donde la lógica suele fallar. Puntos de inflexión (0, null, vacíos, máximos).

Ejemplo

// Factorial de 0 es 1
factorial(0) → 1

// Lista vacía size = 0

Caso de Error

Entradas inválidas que el sistema debe rechazar explícitamente, generalmente lanzando una Excepción.

Ejemplo

// No existe factorial negativo
factorial(-1)
throws IllegalArgumentEx

Las 3 Reglas de Oro

1
Code the Unit Test First TDD

Antes de escribir código funcional, escribe la prueba.
Ciclo: Rojo (Falla) → Verde (Pasa) → Refactor (Mejora).

2
All Code Must Have Tests Zero Tolerance

No se entrega ni se hace commit de código sin pruebas asociadas.
La cobertura debe ser total para la lógica de negocio crítica.

3
Found a Bug? Create a Test! Regression Shield

Si reportan un error, primero crea una prueba que replique el fallo (rojo).
Solo entonces arregla el código. Esto evita que el error vuelva a ocurrir.

TDD + Edge Cases + Debugging
1
TDD First Test
RED GREEN

Testear comportamiento antes de implementar. shouldAumentarSaldo...

@Test void shouldAumentarSaldoWhenConsignarValido() {

  cuenta.consignar(100.0); assert(100.0, cuenta.getSaldo());

}

2
Casos Nominales
HAPPY PATH

El escenario ideal: consignar un monto válido aumenta el saldo correctamente.

public void consignar(double m) {

  this.saldo += m; // Implementación inicial

}

3
Caso de Borde
EDGE CASE

¿Qué pasa si consigno 0? El saldo no debería cambiar. shouldNotAlterar...

@Test void shouldNotAlterarSaldoWhenConsignarCero() {

  cuenta.consignar(0.0); assert(50.0, cuenta.getSaldo());

}

4
Bug Fixing
BUG FOUND

Bug: Consignar negativo resta saldo. Solución: if (m > 0)

@Test void shouldNotPermitirMontoNegativo() {

  cuenta.consignar(-20.0); assert(100.0, cuenta.getSaldo());

}
